 ¿Cómo manejan la conexión a la BD y qué pasa si algunos de los datos son incorrectos? Justifiquen la manera de validación de la conexión.
En nuestro proyecto la conexión a la base de datos se centraliza en conexion.php, donde se usa mysqli con los parámetros de servidor, usuario, contraseña y nombre de la base de datos. Si alguno de esos datos es incorrecto, mysqli genera un connect_error y el sistema detiene la ejecución con die("Error de conexión..."), lo que evita que la aplicación continúe trabajando con una conexión inválida. Además, en login.php se valida si el usuario dejó campos vacíos y también se controla si el usuario no existe o si la contraseña es incorrecta. En facturacion.php también se valida que el JSON recibido sea válido y que campos como Id_producto y Nombre estén presentes antes de insertar datos. Por eso, consideramos que la validación actual es básica pero funcional, ya que primero comprueba que la conexión exista y luego verifica que los datos mínimos requeridos estén correctos antes de operar.

¿Cuál es la diferencia entre $_GET y $_POST en PHP? ¿Cuándo es más apropiado usar cada uno? Da un ejemplo real de tu proyecto.
La diferencia principal es que $_GET recoge variables que viajan en la URL como parte del query string, mientras que $_POST recoge datos enviados en formularios con método POST. Además, la documentación oficial de PHP aclara que cuando se envía JSON no se usa $_POST directamente, sino que debe leerse con php://input, que es justamente lo que hace este proyecto en facturacion.php. Lo más apropiado es usar $_GET cuando solo se desea consultar o identificar un recurso y no se están enviando datos sensibles, mientras que $_POST es mejor cuando se envían credenciales o cuando se crean o actualizan registros. En este proyecto, login.php usa $_POST para recibir nombre y password, y listar.php usa $_POST para actualizar productos desde el formulario de edición. Como ejemplo de $_GET, en facturacion.php se toma Id_producto desde la URL para eliminar un producto.

Tu app va a usarse en una empresa de la zona oriental. ¿Qué riesgos de seguridad identificas en una app web con BD que maneja datos de los usuarios? ¿Cómo los mitigarían?
Uno de los principales riesgos es la inyección SQL, porque en varias partes del proyecto las consultas se construyen concatenando valores directamente dentro del SQL, por ejemplo en el login, en la edición y en las operaciones de productos. Para mitigarlo, usaríamos sentencias preparadas con parámetros (prepared statements y bind_param), ya que OWASP las recomienda como una defensa principal contra este tipo de ataque. El manejo de contraseñas en texto plano tambien es un riesgo importasnte. En el volcado SQL, la tabla usuario guarda la contraseña en un campo password, y en login.php la comparación se hace de forma directa, sin hash. Eso es peligroso porque, si la base de datos se expone, las contraseñas quedarían comprometidas. La mejora correcta sería almacenar contraseñas con password_hash() y validarlas con password_verify().
